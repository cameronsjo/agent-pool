# agent-pool v0.1 Implementation Plan

## Overview

Build the basic expert lifecycle loop: mail arrives in postoffice, daemon parses and routes it to an expert inbox, expert spawns as `claude -p`, output is captured to logs. Six build steps, each depending on the previous.

## Design Decisions

### Q1: Should the router live in internal/mail/ or internal/daemon/?

**Answer: internal/mail/.** The mail package already owns parsing. Routing is a mail-layer concern (read `to:` header, copy file, delete original). The daemon calls `mail.Route()` but does not implement the copy/delete logic. This keeps the daemon thin (orchestration only) and the mail package self-contained. The daemon passes a `poolDir` to the router so it knows where expert inboxes live.

### Q2: How to handle partial file writes in fsnotify?

**Answer: Poll-after-create with a short delay and size stability check.** When fsnotify fires a Create event, wait 100ms, then stat the file. If the size changed, wait another 100ms. Repeat up to 5 times (500ms total max). This handles editors and scripts that write in chunks. An alternative (rename-based atomic writes) could be documented as the recommended pattern for producers, but the daemon must be defensive. The stability check lives in the daemon's watcher loop, not in the mail package.

### Q3: Should inbox watching be part of the daemon or the expert package?

**Answer: The daemon watches inboxes, the expert package handles spawning.** The daemon owns all fsnotify watchers (postoffice + per-expert inboxes) because it needs a unified event loop with context-based shutdown. When an inbox event fires, the daemon calls into `expert.Spawn()`. The expert package is purely about prompt assembly and process management -- it has no awareness of fsnotify.

### Q4: How should we structure the daemon's Run function?

**Answer: Multiple goroutines with a single coordination select.** Structure:

- Main goroutine: sets up watchers, starts the event-processing goroutine, blocks on signal
- Postoffice goroutine: reads fsnotify events, calls mail.Route()
- Per-expert inbox processing: serial within each expert (mutex), but different experts can run concurrently in the future. For v0.1, a single goroutine processes all inbox events sequentially (simplest correct implementation)
- Signal handler goroutine: SIGTERM/SIGINT cancels context

The Run function signature: `func Run(ctx context.Context, cfg *config.PoolConfig, poolDir string) error`

### Q5: How to handle "one at a time per expert"?

**Answer: A `map[string]bool` busy flag guarded by a single `sync.Mutex`.** Before spawning, check if the expert is busy. If so, skip (the file stays in inbox for later). After the spawn completes, clear the flag and drain any remaining inbox files. For v0.1 this is sufficient since inbox events are processed sequentially in a single goroutine anyway. The flag becomes load-bearing in future versions when inbox events may be processed concurrently.

## Build Order

### Step 1: Mail Parser -- `internal/mail/mail.go`

**New dependency:** `gopkg.in/yaml.v3`

**Types:**

```go
// MessageType enumerates the kinds of mail messages.
type MessageType string

const (
    TypeTask     MessageType = "task"
    TypeQuestion MessageType = "question"
    TypeResponse MessageType = "response"
    TypeNotify   MessageType = "notify"
    TypeHandoff  MessageType = "handoff"
    TypeCancel   MessageType = "cancel"
)

// Priority enumerates message urgency levels.
type Priority string

const (
    PriorityLow    Priority = "low"
    PriorityNormal Priority = "normal"
    PriorityHigh   Priority = "high"
    PriorityUrgent Priority = "urgent"
)

// Message represents a parsed mail file (YAML frontmatter + markdown body).
type Message struct {
    // Header fields (from YAML frontmatter)
    ID        string      `yaml:"id"`
    From      string      `yaml:"from"`
    To        string      `yaml:"to"`
    Type      MessageType `yaml:"type"`
    Contracts []string    `yaml:"contracts,omitempty"`
    Priority  Priority    `yaml:"priority"`
    DependsOn []string    `yaml:"depends-on,omitempty"`
    Timestamp time.Time   `yaml:"timestamp"`

    // Body is the markdown content after the frontmatter.
    Body string `yaml:"-"`

    // SourcePath is the file this message was parsed from.
    SourcePath string `yaml:"-"`
}
```

**Functions:**

```go
// ParseFile reads a markdown file with YAML frontmatter and returns a Message.
func ParseFile(path string) (*Message, error)
```

**Implementation notes:**

- Read entire file, split on `---\n` boundaries. Content between first and second `---` is YAML. Everything after second `---` is Body.
- Use `yaml.Unmarshal` for the frontmatter.
- Validate required fields: ID, From, To, Type must be non-empty.
- Default Priority to `"normal"` if empty.
- Set SourcePath to the input path.
- Trim leading/trailing whitespace from Body.

**Test file:** `internal/mail/mail_test.go`

Test cases: valid message with all fields, missing required field returns error, empty body is OK, malformed YAML returns error, unknown message type still parses (no enum validation in v0.1).

### Step 2: fsnotify Watcher -- `internal/daemon/watcher.go`

**New dependency:** `github.com/fsnotify/fsnotify`

**Types and functions:**

```go
// WatcherEvent carries a validated file event.
type WatcherEvent struct {
    Path string // full path to the new file
    Dir  string // which watched directory this came from
}

// Watcher watches directories for new .md files and emits events
// after verifying the file is fully written.
type Watcher struct {
    watcher *fsnotify.Watcher
    events  chan WatcherEvent
    logger  *slog.Logger
}

// NewWatcher creates a Watcher.
func NewWatcher(logger *slog.Logger) (*Watcher, error)

// Add registers a directory to watch.
func (w *Watcher) Add(dir string) error

// Events returns the channel of validated file events.
func (w *Watcher) Events() <-chan WatcherEvent

// Run processes raw fsnotify events until ctx is cancelled.
// Filters for Create events on .md files, applies partial-write
// stability check before emitting.
func (w *Watcher) Run(ctx context.Context)

// Close shuts down the underlying fsnotify watcher.
func (w *Watcher) Close() error
```

**Partial-write handling** inside Run:

```go
func waitForStable(path string, interval time.Duration, maxAttempts int) error
```

Default: 100ms interval, 5 attempts. On each attempt, stat the file and compare size to previous stat. If size is unchanged, the file is stable. If the file disappears, log warning and skip.

**Test file:** `internal/daemon/watcher_test.go`

Integration test: create temp dir, start watcher, write a .md file atomically, assert event received. Verify .txt files are ignored.

### Step 3: Mail Router -- `internal/mail/router.go`

**Functions:**

```go
// ResolveInbox returns the inbox path for a recipient within a pool.
//
// Built-in roles (architect, researcher, concierge):
//   {poolDir}/{role}/inbox/
//
// Experts:
//   {poolDir}/experts/{name}/inbox/
func ResolveInbox(poolDir, recipient string) string

// Route parses a message from postoffice, copies it to the
// recipient's inbox, and deletes the original.
// Delivery is at-least-once: copy completes before delete.
func Route(logger *slog.Logger, poolDir, filePath string) (*Message, error)
```

**Implementation notes:**

- `Route` calls `ParseFile` to get the `To` field.
- `ResolveInbox`: if recipient is `"architect"`, `"researcher"`, or `"concierge"`, use `{poolDir}/{role}/inbox/`. Otherwise `{poolDir}/experts/{recipient}/inbox/`.
- Copy: write to `{inbox}/{msg.ID}.md` (use message ID as filename, not the source filename, to avoid collisions from duplicate delivery).
- Delete original only after successful write.
- Return error if inbox directory does not exist.

**Test file:** `internal/mail/router_test.go`

Integration test: create temp pool structure with expert inbox dirs, write a message to fake postoffice, call Route, verify file appears in inbox and is gone from postoffice. Test unknown recipient error.

### Step 4: Expert Spawner -- `internal/expert/expert.go`

**Types:**

```go
// SpawnConfig holds everything needed to spawn an expert session.
type SpawnConfig struct {
    Name         string
    Model        string
    AllowedTools []string

    ExpertDir    string        // contains identity.md, state.md, errors.md

    TaskID   string

    Summary  string         // extracted final result text

}
```

**Functions:**

```go
// AssemblePrompt reads state files + task and builds the prompt string.
// Order: identity.md, state.md, errors.md, task body + metadata.
// Missing state files are silently skipped.
func AssemblePrompt(cfg *SpawnConfig) (string, error)

// Spawn starts a claude -p session and blocks until it exits.
// Command: claude -p --output-format stream-json --model {model} --allowedTools "{tools}"
// Prompt is written to stdin. Stdout is captured.
// Environment: AGENT_POOL_EXPERT, AGENT_POOL_TASK_ID
func Spawn(ctx context.Context, logger *slog.Logger, cfg *SpawnConfig) (*Result, error)
```

**AssemblePrompt format:**

```markdown
## Expert Identity

{identity.md contents}

## Current State

{state.md contents}

## Known Errors & Pitfalls

{errors.md contents}

## Task

{task body}

### Task Metadata
- ID: {id}
- From: {from}
- Type: {type}
- Priority: {priority}
- Contracts: {comma-joined, or "none"}
```

Sections with missing files are omitted entirely (not rendered with empty content).

**Spawn implementation notes:**

- Use `exec.CommandContext` for cancellation support.
- Set `cmd.Dir = cfg.ProjectDir` (expand ~ first).
- Set `cmd.Stdin = strings.NewReader(prompt)`.
- Capture stdout via `cmd.Output()` or pipe + read.
- Log stderr lines via slog.
- Set env vars: `AGENT_POOL_EXPERT`, `AGENT_POOL_TASK_ID`.
- Record start time, compute duration after Wait().
- Check `exec.LookPath("claude")` before spawning, return clear error if missing.

- Unit test for AssemblePrompt: create temp expert dir with state files, verify output.
- Integration test for Spawn: guard with `exec.LookPath("claude")`, skip if unavailable.

### Step 6: Daemon Run Loop -- `internal/daemon/daemon.go`

**Types:**

```go
type Daemon struct {
    cfg     *config.PoolConfig
    poolDir string
    logger  *slog.Logger
    watcher *Watcher

    mu   sync.Mutex
    busy map[string]bool
}
```

**Functions:**

```go
func New(cfg *config.PoolConfig, poolDir string, logger *slog.Logger) *Daemon
func (d *Daemon) Run(ctx context.Context) error
func (d *Daemon) ensureDirs() error
func (d *Daemon) handlePostoffice(ctx context.Context, path string)
func (d *Daemon) handleInbox(ctx context.Context, expertName string, path string)
func (d *Daemon) resolveExpertConfig(expertName string) (model string, tools []string)
func (d *Daemon) drainInbox(ctx context.Context, expertName string)
```

**Run loop:**

1. `ensureDirs()` -- create postoffice/, expert inboxes, logs dirs
2. Create watcher, add postoffice dir + all expert inbox dirs
3. Start `watcher.Run(ctx)` in goroutine
4. Select loop on `watcher.Events()` and `ctx.Done()`
5. Dispatch events: postoffice events call `handlePostoffice`, inbox events call `handleInbox`
6. On `ctx.Done()`, return nil

**handlePostoffice:** calls `mail.Route()`, logs result.

**handleInbox:**
1. Check busy flag, skip if busy (log warning, file stays for later)
2. Set busy = true
3. Parse message, resolve config, call `expert.Spawn()`
4. Write logs (WriteLog + AppendIndex)
5. Delete processed inbox file
6. Set busy = false
7. Call `drainInbox` to pick up any files that arrived while busy

**drainInbox:** list .md files in expert's inbox, sort by mod time, process oldest one (recursive through handleInbox). This ensures queued messages are not lost.

**resolveExpertConfig:** look up expert in `cfg.Experts[name]`. Use expert's model if set, otherwise `cfg.Defaults.Model`. Same for tools.

**Wire into main.go:**

```go
func cmdStart() {
    // ... existing config loading ...

    logger := slog.New(slog.NewJSONHandler(os.Stdout, &slog.HandlerOptions{
        Level: slog.LevelInfo,
    }))

    ctx, stop := signal.NotifyContext(context.Background(),
        syscall.SIGTERM, syscall.SIGINT)
    defer stop()

    d := daemon.New(cfg, poolDir, logger)
    if err := d.Run(ctx); err != nil {
        logger.Error("daemon failed", "error", err)
        os.Exit(1)
    }
}
```

Note: `poolDir` must be resolved to an absolute path before passing to the daemon. The current `cmdStart` gets it from `os.Args[2]` or uses cwd. Need to `filepath.Abs()` it.

## Edge Cases & Hazards

1. **ProjectDir tilde expansion.** `config.LoadPool` does not expand `~` in `Pool.ProjectDir`. The daemon must expand it before passing to expert.Spawn as `cmd.Dir`.

2. **Empty expert config.** `resolveExpertConfig` must fall back to `cfg.Defaults.Model` and `cfg.Defaults.AllowedTools` when the expert section has empty values.

3. **Missed inbox events.** If a message arrives in an inbox while the expert is busy, the fsnotify event fires but handleInbox skips it. The `drainInbox` call after each spawn completion handles this. Also, on daemon startup, drain all inboxes to pick up files that arrived while the daemon was down.

4. **claude not in PATH.** `Spawn` should call `exec.LookPath("claude")` early and return `fmt.Errorf("claude binary not found in PATH: %w", err)`.

5. **stream-json format.** The exact output shape of `--output-format stream-json` needs validation during smoke testing. ExtractSummary must be defensive against unknown JSON structures.

6. **Inbox filename collisions.** Route uses `{msg.ID}.md` as the inbox filename. If a duplicate delivery occurs (at-least-once), it overwrites the previous copy, which is fine (same content).

7. **Watcher overflow.** fsnotify has a kernel-level event queue. If the daemon is blocked processing a long expert spawn and many files arrive, events may be dropped. The startup drain and post-spawn drain mitigate this.
